A camada do comportamento.

JS roda em uma única thread

console.log('1 — antes');
setTimeout(() => console.log('3 — depois de 1s'), 1000);
console.log('2 — depois');
// 1 → 2 → 3
Operações lentas não bloqueiam a página: o JS agenda e continua.
Single-thread + event loop. Se você travar a thread, a interface congela. Por isso quase tudo que espera é assíncrono.

Tipos e comparação

const nome = 'Ana';   // string   (não reatribui)
let idade = 20;       // number   (reatribui)
const ativo = true;   // boolean

0 == '0'    // true  😱  (coerção)
0 === '0'   // false ✔️  (compara tipo também)
Use sempre ===. Prefira const; let só quando reatribuir.

Funções

function soma(a, b) { return a + b; }

const dobro = n => n * 2;   // arrow function

Funções são valores: podem ser passadas e retornadas.

O trio que abre a cabeça

const nums = [1, 2, 3, 4];

nums.map(n => n * 2);        // [2, 4, 6, 8]
nums.filter(n => n % 2 === 0); // [2, 4]
nums.reduce((a, n) => a + n, 0); // 10

map transforma, filter seleciona, reduce acumula.

Substituem a maioria dos loops for. É o salto para pensar de forma declarativa.

Não trave a página esperando.

Assincronia.

Da callback à Promise

// callback hell 😖
buscarUsuario(id, (e, u) => {
  buscarPedidos(u.id, (e, p) => {
    buscarItens(p[0], (e, i) => { /* ... */ });
  });
});

Aninhamento profundo → ilegível. As Promises resolvem isso.

Promise tem 3 estados: pending, fulfilled, rejected. .then encadeia, .catch trata.

async/await: assíncrono que lê como síncrono

async function carregar() {
  try {
    const u = await buscarUsuario(id);
    const p = await buscarPedidos(u.id);
    return p;
  } catch (e) {
    console.error(e);
  }
}

A página é uma árvore viva.

DOM.

Selecionar com o CSS que você já sabe

const titulo = document.querySelector('h1');
const itens  = document.querySelectorAll('.item');

O mesmo seletor do CSS — nada novo para aprender.

Alterar (com cuidado)

titulo.textContent = 'Olá';      // ✔️ seguro
caixa.innerHTML = dadosDoUsuario; // ⚠️ risco de XSS
innerHTML com dado não confiável = XSS. Prefira textContent.

Reagir a eventos

botao.addEventListener('click', () => {
  contador++;
  placar.textContent = contador;
});

O ciclo: selecionar → alterar → ouvir.

Propagação e delegação

captura ↓→ alvo→ bubbling ↑
Delegação: um listener no pai atende muitos filhos — inclusive os criados depois.
Em vez de N listeners, um só no container e checa event.target. Essencial para listas dinâmicas.

Clique de verdade

Clique no botão à direita durante a apresentação — o número muda ao vivo. É o ciclo selecionar → ouvir evento → alterar, funcionando de verdade no iframe.

Delegação: um listener, muitos itens

Clique nos itens — cada um alterna o destaque, mas há só UM listener no <ul>. Funcionaria até para itens adicionados depois.

Validação com JavaScript

if (!email.validity.valid) {
  email.setCustomValidity('E-mail inválido');
}

Constraint Validation API: valida em tempo real, com feedback claro.

Continua valendo: validação no cliente é UX, não segurança. A definitiva é no servidor.

Validação em tempo real

Digite no campo à direita — a borda e a mensagem mudam a cada tecla, usando o evento input e a Constraint Validation API (email.validity.valid). Lembre: isto é UX; a validação de segurança é no servidor.

O padrão fetch robusto

async function buscar(url) {
  const r = await fetch(url);
  if (!r.ok) throw new Error(`HTTP ${r.status}`);
  return r.json();
}

await fetch → checa ok → await json().

⚠️ A pegadinha nº 1

fetch não rejeita em erro HTTP (404, 500)! Só falha em erro de rede. Por isso sempre cheque resposta.ok.
Bug clássico: fazer .then(r => r.json()) direto e nunca detectar o 404. O 404 chega como Response normal.

Toda tela que carrega tem 4 estados

carregando→ sucesso
vazio· erro

Trate todos — não deixe o usuário olhando uma tela em branco.
