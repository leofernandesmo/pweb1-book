A Web não é a Internet.

Pausa. Deixe a provocação no ar antes de explicar. É o erro conceitual mais comum.

Internet × Web

InternetWeb
Infraestrutura físicaSistema de informação
Cabos, roteadores, TCP/IPDocumentos ligados por hiperlinks
O "caminho"O que trafega sobre o caminho
A Web é uma camada de hipermídia sobre a Internet: recursos identificados por URIs e ligados por hiperlinks.
Analogia: a Internet é a malha rodoviária; a Web é um dos serviços que rodam nela (e-mail, streaming e jogos online são outros).

Arquitetura cliente-servidor

Cliente— requisição → Servidor
Cliente← resposta — Servidor

Três papéis para separar (setas ↓ para aprofundar).

Cliente = quem inicia. Servidor = quem escuta e responde. O navegador é um tipo específico de cliente. Use as setas para baixo para detalhar cada um.

Cliente

Software que inicia a comunicação (o User Agent).

Navegador, app móvel, curl, bots/crawlers.

Servidor

Software que escuta portas (80/443) e responde.

Apache, Nginx, IIS — rodando como processo em segundo plano.

Navegador

Cliente com motor de renderização: transforma HTML/CSS/JS em pixels.

Blink (Chrome/Edge), Gecko (Firefox), WebKit (Safari).

HTTP: a conversa

Texto, sem estado (stateless), em pares requisição → resposta.

Request→ servidor processa→ Response
Stateless = cada requisição é independente; o servidor não "lembra" da anterior. Isso é o que permite a Web escalar. Cookies/tokens existem justamente para recriar esse estado quando ele é necessário.

Uma requisição

GET /produtos HTTP/1.1
Host: www.exemplo.com
User-Agent: Mozilla/5.0
Accept: text/html

Linha de requisição + cabeçalhos (+ corpo, em POST/PUT).

Uma resposta

HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 1024

<!doctype html> ...

Linha de status + cabeçalhos + corpo (o recurso).

Códigos de status

ClasseSignificadoExemplos
2xxSucesso200, 201
3xxRedirecionamento301, 302
4xxErro do cliente400, 404
5xxErro do servidor500, 503
Regra prática: 4xx = "você errou o pedido"; 5xx = "o servidor falhou".
Pergunte à turma onde encaixa um 404 (cliente pediu algo que não existe) e um 500 (bug no servidor). Mencione 1xx só de passagem.

O que acontece quando você digita uma URL?

A pergunta que junta tudo.
Este é o clímax da unidade. Peça para a turma tentar responder antes de avançar.

exemplo.com/produtos → tela

cache?→ DNS→ TCP/QUIC→ HTTP→ render

Tudo isso em milissegundos (seta ↓ para DNS e render).

Passe rápido pelos passos: verifica cache local; resolve o nome via DNS; abre conexão; envia a requisição; recebe a resposta; renderiza.

DNS: o nome vira número

www.exemplo.com→ 192.0.2.1

Cadeia hierárquica: Root → TLD → Autoritativo. UDP porta 53.

Renderização

  1. Lê o HTML → monta o DOM
  2. Baixa CSS, JS e imagens
  3. Aplica estilos e executa scripts
  4. Exibe a página

Ferramentas do curso

  • DevTools — inspecionar e depurar (F12)
  • VS Code — editor
  • Git + GitHub — versão e entrega
  • CodePen — testar no navegador
DevTools transforma o navegador em laboratório: DOM ao vivo, CSS, rede, performance, dispositivos.
Abra o DevTools ao vivo numa página qualquer (aba Elements e Network). Nada convence mais que ver o HTML real de um site conhecido.

Estrutura de um projeto

meu-projeto/
├── index.html
├── css/
│   └── style.css
├── js/
│   └── script.js
└── assets/
    └── images/

Separar por tipo, nomes em kebab-case, sem código inline.

Convenção simples desde o primeiro exercício reduz atrito e prepara para trabalhar em repositório. Mostraremos isso na prática na Unidade 1.

Resumo

  • A Web roda sobre a Internet.
  • Cliente-servidor via HTTP (stateless).
  • Status: 2xx/3xx/4xx/5xx.
  • URL → DNS → conexão → HTTP → render.
  • DevTools, VS Code, Git/GitHub são o dia a dia.
