Capítulos 2 a 6 do livro.

HTML descreve o quê, não o como.

Estrutura e significado — a aparência é do CSS.
O erro-raiz que a unidade combate: escolher tag pelo tamanho visual. Título é título porque é título — não porque "ficou grande".

Todo documento começa assim

<!DOCTYPE html>
<html lang="pt-BR">
  <head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1">
    <title>Título da aba</title>
  </head>
  <body>
    <p>Conteúdo visível.</p>
  </body>
</html>
lang e charset="UTF-8" são obrigatórios na prática: impactam acessibilidade e acentuação.
DOCTYPE liga o standards mode. head = metadados (não aparece). body = o que se vê.

Tag, elemento e atributo

<a href="https://ifal.edu.br">Site do IFAL</a>
  • Tag — o delimitador: <a> … </a>
  • Elemento — tag de abertura + conteúdo + fechamento
  • Atributo — href="...", informação extra
Vocabulário preciso evita confusão na documentação. Void elements (img, br) não têm fechamento.

O navegador constrói o DOM

document
└── html
    ├── head → title → "..."
    └── body → p → "Conteúdo"

Árvore de nós em memória — é o que o CSS e o JS enxergam.

Significado > aparência

SemânticoSó visual
<strong> importância<b> negrito
<em> ênfase<i> voz alternativa
Leitores de tela mudam a entonação em <strong>/<em> — mas não em <b>/<i>.
"É obrigatório fazer backup" vs. só deixar em negrito: o primeiro comunica urgência à tecnologia assistiva.

Links: o coração da Web

<a href="https://externo.com"
   target="_blank"
   rel="noopener noreferrer">Texto descritivo</a>
  • target="_blank" exige rel="noopener" (segurança).
  • Texto do link deve fazer sentido fora de contexto — nunca "clique aqui".
Usuários de leitor de tela navegam por uma lista de links. "Clique aqui" repetido 10× é inútil.

Imagem sem alt é bug

<img src="grafico.png"
     alt="Vendas cresceram 40% no 2º semestre">

<img src="ornamento.svg" alt="">  <!-- decorativa -->

alt descreve função, não é legenda. Decorativa: alt="" (vazio, não ausente).

Legenda visível? Use <figure> + <figcaption>.

Três listas, três sentidos

<ul>

ordem irrelevante

<ol>

ordem importa (passos, ranking)

<dl>

termo → descrição

Aninhar sempre dentro de <li> — nunca <ul> direto dentro de <ul>.

Chega de <div> para tudo.

"divitis" — o anti-padrão nº 1.

O mapa semântico

<body>
├── <header>   cabeçalho + <nav>
├── <main>     conteúdo único da página
│   ├── <article>  autônomo
│   │   └── <section> tema
│   └── <aside>    tangencial
└── <footer>   rodapé

Sem efeito visual automático — só significado. O CSS cuida do layout.

Cada região comunica sua função ao navegador, ao Google e ao leitor de tela. main é único por página.

article × section × div

  • article — faz sentido sozinho (post, comentário)
  • section — parte de um todo, tem título
  • div — só agrupamento visual, sem semântica

Antes → depois

❌ div-soup

<div class="cabecalho">
 <div class="titulo">Blog</div>
 <div class="menu">...</div>
</div>
<div class="post">...</div>

✔️ semântico

<header>
 <h1>Blog</h1>
 <nav>...</nav>
</header>
<article>...</article>

Mesmo visual. Estrutura radicalmente melhor.

Refatorar não muda o visual — muda o significado. Cada troca tem um critério.

Mito: "algoritmo de outline"

Nenhum navegador implementou a geração automática de outline por <section>. A hierarquia vem direto dos níveis <h1>–<h6>. Defina o nível você mesmo; prefira um <h1> por página.

Tabela = dados, não layout

<table>
  <caption>Horário</caption>
  <thead>
    <tr><th scope="col">Disciplina</th><th scope="col">Dia</th></tr>
  </thead>
  <tbody>
    <tr><td>PWeb 1</td><td>Sexta</td></tr>
  </tbody>
</table>

caption, th scope, thead/tbody = tabela acessível.

Nunca use tabela para posicionar a página — leitor de tela anuncia "tabela, N linhas" e a navegação vira um inferno.

Layout é do CSS

Se não é uma grade de dados, não é tabela. Use <div> + CSS (Flexbox/Grid — Unidade 2).

Imagens responsivas

  • srcset — várias resoluções, o navegador escolhe
  • sizes — quanto a imagem ocupa por viewport
  • <picture> — trocar a imagem (formato/enquadramento)
srcset/sizes = mesma imagem em tamanhos; picture = imagem diferente (AVIF/WebP com fallback, ou art direction).

video, iframe e acessibilidade

  • <track> — legendas WebVTT (obrigatório p/ WCAG)
  • <iframe> — title obrigatório + loading="lazy"
  • Prefira youtube-nocookie.com em material educacional

Formulários coletam dados. Não os processam.

Quem processa é o backend.

<form>: para onde e como

<form action="/cadastro" method="post"> ... </form>
GETPOST
Dadosna URLno corpo
Usobusca, filtrologin, cadastro

Dados sensíveis? Nunca GET.

httpbin.org é o laboratório: envie e veja o JSON do que chegou.

name × id

Sem name, o campo não é enviado — mesmo preenchido e válido. É o bug mais comum de iniciante.

id = identidade no documento (label, CSS, JS). name = identidade no envio.

O type muda tudo

textemail passwordnumber daterange filecheckbox radio

Um atributo define teclado móvel, validação e interface nativa.

radio (mesmo name) = escolha única; checkbox = zero ou várias.

Formulário acessível

<fieldset>
  <legend>Dados pessoais</legend>
  <label for="nome">Nome</label>
  <input id="nome" name="nome" required>
</fieldset>
  • <label for> em todo campo — placeholder não substitui.
  • <fieldset>+<legend> agrupam (essencial p/ radio).

Validação no navegador não é segurança.

É conveniência. A definitiva é no servidor.
required/pattern melhoram a UX, mas o DevTools contorna tudo. Repita isso à exaustão.

Acessibilidade beneficia todos

1,3 bilhão de pessoas (OMS) com deficiência. E o resto também ganha:

Legenda no barulho, contraste no sol, teclado por velocidade — o curb-cut effect.

Rampa de calçada foi feita p/ cadeirante e virou útil p/ carrinho, bike, mala. Acessibilidade é assim. No Brasil ainda é lei (LBI 13.146/2015).

POUR — os 4 princípios

  • Perceptível — dá pra perceber (alt, legenda, contraste 4,5:1)
  • Operável — dá pra usar só com teclado
  • Understandable — compreensível (lang, erros claros)
  • Robusto — HTML válido e semântico

Primeira regra do ARIA

Não use ARIA se existe elemento nativo. <button> > <div role="button">. ARIA complementa — não substitui HTML semântico.

Checklist de toda entrega

  • lang no <html>
  • Um <h1>, sem pular níveis
  • alt em toda imagem
  • Contraste ≥ 4,5:1
  • Navegável por teclado
  • <label> em todo campo

Ferramentas: W3C Validator → DevTools Accessibility → WAVE → Lighthouse.

Resumo

  • HTML = estrutura e significado.
  • Escolha o elemento pelo sentido, não pelo visual.
  • Tabela = dados; layout é do CSS.
  • Formulário coleta; backend processa e valida.
  • Acessibilidade desde o início (POUR).
